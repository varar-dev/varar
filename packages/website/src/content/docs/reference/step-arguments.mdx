# Step argument types

A step handler is `(state, ...args) => …`. The first parameter, `state`, is your
state — its type comes from `defineState`. Everything after it is captured from
the `{…}` **parameters** in the cucumber expression, and Vár infers those types
directly from the expression string. **You don't annotate them.**

```ts
const { action } = defineState(() => ({ greeting: '' }))

// `name` is inferred as string — no annotation needed.
action('I greet {string}', (state, name) => {
  // context/action steps EVOLVE state by RETURNING a partial — never by
  // mutating. The returned object is shallow-merged onto the current state.
  return { greeting: `Hello, ${name}!` }
})
```

## Built-in parameter types

Each built-in cucumber parameter type maps to a TypeScript type:

| Parameter type                                                          | Inferred type |
| ----------------------------------------------------------------------- | ------------- |
| `{int}` `{float}` `{double}` `{long}` `{short}` `{byte}` `{bigdecimal}` | `number`      |
| `{biginteger}`                                                          | `bigint`      |
| `{string}` `{word}` `{}` (anonymous)                                    | `string`      |

Parameters are inferred in source order, so `'expression {int}+{int}'` gives a
handler `(state, a, b)` where both `a` and `b` are `number`. Here is the whole
`01-hello` step file — note that **no parameter is annotated** — running against
its spec:

A wrong annotation is still an error: `{int}` infers `number`, so writing
`(state, count: string)` fails to type-check. Annotate only when you want to
**narrow** a slot the inference can't reach (below) — never to restate what the
expression already says.

## Custom parameter types

Declare custom parameter types in the second argument to `defineState`, and
their transformer's **return type** is inferred for every matching placeholder —
just like the built-ins. No annotations:

```ts
const { context } = defineState(() => ({ member: 'maya' }), {
  date: { regexp: /…/, transformer: (m: string, d: string) => new Date(/* … */) },
})

// {date} → Date, {string} → string, both inferred.
context('borrowed {string}, due back on {date}', (_state, title, due) => {
  // `title` is string, `due` is Date.
})
```

The transformer is the source of truth: `(…) => Date` makes `{date}` a `Date`,
`(…) => number` makes it a `number`. Because the declaration also registers the
type for matching, you no longer need a separate `defineParameterType` call.

> The standalone `defineParameterType(…)` function still works for registering a
> type globally, but a placeholder backed by it falls back to `any` (the
> registration isn't connected to the step at the type level). Declare types in
> `defineState` to get inference.

## The trailing table or doc-string argument

This is the one slot inference can't reach. A data table or doc string attached
to a step arrives as the **last** argument, *after* the captured parameters. It
has no `{…}` placeholder in the expression, so it is inferred as `any` — annotate
it. See [Tables](/docs/reference/tables) and
[Doc strings](/docs/reference/doc-strings) for the shapes.

```ts
// `name` inferred from {word}; `body` is the trailing doc string — annotate it.
sensor('Greet {word}:', (_state, name, body: string) => [name, `Hello, ${name}!\n`])
```

## Escaped braces

A brace escaped with a backslash is literal text, not a parameter, so it
produces no argument. `'a literal \{brace\} and an {int}'` yields a single
`number` argument.
