# Go port — task plan

Date: 2026-07-19
Status: in progress

Implements the Go port per [ADR 0010](../../adr/0010-go-port.md),
[ADR 0011](../../adr/0011-go-test-integration.md), and the two design specs
([core](../specs/2026-07-19-go-core-port-design.md),
[runner+adapter](../specs/2026-07-19-go-runner-adapter-design.md)). Each core
stage is gated by reproducing the shared goldens byte-for-byte before the next.

## Follow-ups

- [x] **Plain Go handler parameters** on `Sensor`/`Stimulus`: each slot arrives
      as a plain Go value and a sensor returns one value per slot, mirroring the
      TS tuple return. Done with reflection in the facade (core stays
      reflection-free), validated eagerly at registration. A generic
      `Sensor1/2/3` variant was prototyped first and dropped: it checks more at
      compile time, but Go has no variadic generics so the arity must live in
      the name, which is not a Go idiom. The raw `[]Value` form stays as the
      escape hatch. See [ADR 0010](../../adr/0010-go-port.md).
- [x] **Minimise the author-facing API surface** (per the skill's "keep the
      public API surface minimal"): `Steps[C]` for the state, plain Go slot
      types, `Param` keyed on the author's own type, `[][]string` tables,
      `map[string]string` rows, and automatic struct↔value mapping. The example
      project now mentions the dynamic value type **zero** times, down from 123.
- [ ] Update the website docs' Go tabs — they still show the old `[]Value`
      handler shape and `varar.Value` state.
- [ ] **Apply the same minimisation to Rust** (and review C#): its facade
      re-exports `Value` and its handlers take/return it, so a Rust step file
      mentions the dynamic value type everywhere Go no longer does. The same
      moves apply — a context type parameter on the builder, plain Rust slot
      types, and a `param` keyed on the author's own type. Needs its own ADR
      amendment + plan; Rust is already shipped and conformance-green, so this
      is a deliberate API break, not a bug fix.
